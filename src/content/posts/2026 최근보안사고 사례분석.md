## 1. 2025 SK텔레콤 유심 해킹 사건

### 사건 개요

2025년 4월 18일, 대한민국 1위 이동통신사인 SK텔레콤에서 고객 유심(USIM) 정보가 대량 유출된 사실이 공개되었다. 유출된 데이터는 **9.82GB** 규모로, 가입자 식별키(IMSI) 기준 **2,695만 7,749건**에 달한다. 이는 국내 통신 역사상 최대 규모의 개인정보 침해 사례로 기록되었다.

| 항목 | 내용 |
|------|------|
| 발생 일시 | 2025년 4월 18일 (공개) |
| 피해 규모 | 유심 정보 약 2,696만 건 (9.82GB) |
| 유출 정보 | IMSI(가입자 식별키), IMEI(단말기 식별번호), 전화번호, 유심 인증키 등 총 25종 |
| 공격 벡터 | HSS 서버 내 BPFDoor 악성코드 |

### 공격 경위 및 방법

#### 1단계 — Ivanti VPN 취약점 악용

보안 업계에서는 해커가 **이반티(Ivanti) VPN 장비의 제로데이 취약점**을 통해 SK텔레콤 내부 네트워크에 최초 침투했을 가능성을 높게 보고 있다. Ivanti VPN은 2025년 초부터 다수의 제로데이 취약점이 연속 발견되어 전 세계적으로 피해가 속출한 제품이다.

#### 2단계 — HSS 서버 장악

침투에 성공한 공격자는 이동통신 네트워크의 핵심 인프라인 **HSS(Home Subscriber Server)**에 악성코드를 심었다. HSS는 가입자 인증, 위치 정보, 유심 데이터를 관리하는 통신망의 '두뇌'에 해당하는 서버다.

#### 3단계 — BPFDoor 백도어

공격에 사용된 악성코드는 **BPFDoor(Berkeley Packet Filter Door)**로, 기존 보안 솔루션의 탐지를 장기간 우회했다.

- **방화벽 우회:** 특정 포트를 열지 않고 기존 허용된 트래픽에 숨어 동작
- **프로세스 위장:** 정상 프로세스명으로 위장하여 EDR·AV 탐지 회피
- **지속성 유지:** 재부팅 후에도 동작을 유지하는 지속성(Persistence) 메커니즘 보유
- **장기 잠복:** 공격자는 장기간 발각되지 않으며 데이터를 지속적으로 수집

#### 4단계 — 유심 데이터 유출

BPFDoor를 통해 시스템 권한을 확보한 공격자는 HSS에 저장된 유심 관련 데이터를 외부로 유출했다.

- **가입자 식별키(IMSI):** 전 세계 어디서나 가입자를 식별하는 고유 번호
- **단말기 고유식별번호(IMEI):** 스마트폰 하드웨어의 고유 번호
- **유심 인증키(Ki):** 통신사 네트워크와 유심 간 인증에 사용되는 핵심 키값

### 피해의 심각성 — 심 스와핑(SIM Swapping) 위협

단순 개인정보 유출과 달리, 유심 인증키(Ki)가 포함된 이번 유출은 **심 스와핑(SIM Swapping) 공격**을 가능하게 한다는 점에서 특히 심각하다.

> **심 스와핑이란?**  
> 피해자의 전화번호를 공격자가 통제하는 유심으로 이전하는 공격. 성공 시 SMS 인증코드를 가로채 금융 계좌, SNS, 이메일 등 2단계 인증을 모두 탈취 가능하다.

IMSI + Ki 조합이 유출되면 이론적으로 통신사 시스템 없이도 피해자의 유심을 복제할 수 있어, 단순 개인정보 유출 수준을 훨씬 넘어서는 피해가 발생할 수 있다.

### 근본 원인 분석

| 원인 | 내용 |
|------|------|
| 외부 장비 취약점 관리 실패 | Ivanti VPN의 연속 취약점 발견에도 신속한 패치 미적용 |
| 핵심 인프라 접근 통제 미흡 | HSS 서버에 악성코드가 장기 잠복 가능한 허술한 통제 |
| 중요 데이터 암호화 부재 | Ki 등 민감 데이터가 평문으로 저장 |
| 이상 트래픽 탐지 체계 미비 | 9.82GB 대용량 유출을 실시간 탐지 못함 |
| 사고 공개 지연 | 인지 후 72시간 이내 고객 통보 미이행 |

### SK텔레콤의 대응

| 조치 | 내용 |
|------|------|
| 유심 무상 교체 | 2025년 4월 25일 발표, 전 고객 대상 무료 USIM 교체 실시 |
| 유심보호서비스 | 비정상 유심 교체 시 통신 서비스 차단 (홈페이지·T월드 무료 신청) |
| 정부 신고 | 개인정보보호위원회, 과학기술정보통신부 등에 침해 신고 |
| 수사 협조 | 경찰청 사이버수사대, KISA 등과 공조 수사 진행 |

### 미흡했던 점 및 보완 방향

1. **3rd 파티 장비 취약점 스캔 의무화** — Ivanti 취약점은 PoC가 유통 중인 상태였다
2. **핵심 데이터 HSM 암호화** — Ki, IMSI 등을 하드웨어 보안 모듈로 암호화 저장
3. **NDR·UEBA 도입** — 대용량 데이터의 비정상 외부 전송을 실시간 탐지
4. **제로 트러스트 아키텍처** — 내부망도 신뢰하지 않는 보안 패러다임 전환
5. **위기 커뮤니케이션 프로세스** — 사고 인지 즉시 고객 통보 절차 사전 수립

### 교훈

SK텔레콤 유심 해킹 사건은 단순한 기술적 실패가 아닌 **보안 거버넌스 전반의 실패**를 보여준다. 2,696만 명은 대한민국 인구의 절반에 해당하며, 통신 인프라 보안의 중요성을 국가적 차원에서 재점검하는 계기가 되어야 한다.

## 2. 2025 Bybit 가상자산 해킹 사건

### 사건 개요

2025년 2월 21일, 세계 최대 암호화폐 파생상품 거래소 중 하나인 **Bybit**에서 약 **15억 달러(한화 약 2조 1,000억 원)** 상당의 이더리움(ETH)이 도난당했다. 이는 가상자산 역사상 **단일 해킹 사건으로는 최대 규모**이며, FBI는 배후로 북한 해킹 조직 **라자루스 그룹(Lazarus Group, 일명 TraderTraitor)**을 지목했다.

| 항목 | 내용 |
|------|------|
| 발생 일시 | 2025년 2월 21일 |
| 피해 금액 | 약 15억 달러 (이더리움 401,346 ETH) |
| 공격자 | 북한 라자루스 그룹 (TraderTraitor) |
| 공격 방식 | 공급망 공격 (Safe{Wallet} 개발자 PC 침해) + 사회공학 |
| 결과 | 전량 외부 지갑으로 이전, 세탁 진행 중 |

### 공격 경위 — 정교한 공급망 침투

#### 1단계 — Safe{Wallet} 개발자 PC 침해

Bybit은 멀티시그(Multi-Signature) 콜드 월렛 관리를 위해 **Safe{Wallet}**(구 Gnosis Safe) 플랫폼을 사용하고 있었다. 공격자는 Bybit을 직접 공격하는 대신, Safe{Wallet}의 **개발자 한 명의 PC를 먼저 감염**시켜 AWS 자격증명 및 웹사이트 파일 수정 권한을 탈취했다.

#### 2단계 — 악성 JavaScript 코드 삽입

Safe{Wallet} 웹사이트에 **Bybit 지갑 감지 시에만 활성화되는 맞춤형 악성 코드**를 삽입했다. 일반 사용자에게는 정상적으로 동작하므로 탐지가 극히 어려웠다.

#### 3단계 — 트랜잭션 하이재킹

Bybit 담당자가 일상적인 ETH 이체를 승인하는 순간, 악성 코드가 활성화되었다. 화면에는 정상 트랜잭션이 표시되었지만, 실제로 서명된 명령은 **공격자 주소로 전액 이체**하는 것이었다.

#### 4단계 — 자금 세탁

탈취 직후 ETH를 수천 개 지갑으로 분산했다. 2025년 3월 20일 기준 탈취 ETH의 **86.29%가 BTC로 환전**되었으며, 믹서·체인 브리지·P2P 거래를 활용한 세탁이 빠르게 진행되었다.

### 라자루스 그룹이란?

북한 정찰총국(RGB) 산하 국가 지원 해킹 조직으로, FBI에 따르면 탈취 자금을 **핵·미사일 프로그램 자금**으로 활용한다.

| 연도 | 사건 |
|------|------|
| 2014 | 소니 픽처스 해킹 |
| 2016 | 방글라데시 중앙은행 SWIFT 해킹 ($81M) |
| 2017 | 워너크라이(WannaCry) 랜섬웨어 |
| 2022 | Ronin Network 해킹 ($625M) |
| 2025 | **Bybit 해킹 ($1.5B) — 역대 최대** |

### 근본 원인 분석

| 원인 | 내용 |
|------|------|
| 3rd 파티 의존성 리스크 | 외부 플랫폼(Safe{Wallet})에 자산 관리 의존 |
| 트랜잭션 서명 UI 신뢰 취약성 | 화면 표시 데이터와 실제 서명 데이터의 불일치를 검증하지 않음 |
| 멀티시그의 맹점 | 서명자 전원이 동일하게 속으면 다중 서명도 무력화 |
| 개발자 보안 인식 부족 | 개발자 단말 보안 및 코드 배포 파이프라인 무결성 검증 미흡 |

### Bybit의 대응

- 사고 인지 즉시 **출금 일시 중단** 및 내부 조사 착수
- Chainalysis·TRM Labs 등과 협력하여 **자금 추적 개시**
- 주요 거래소에 탈취 지갑 주소 공유, **동결 요청**
- 긴급 자금 확보로 **사용자 자산 전액 보전** 약속 이행

### 미흡했던 점 및 보완 방향

1. **SRI(서브리소스 무결성) 적용** — 3rd 파티 스크립트 해시 검증으로 UI 변조 탐지
2. **독립 디바이스 트랜잭션 검증** — 서명 전 Ledger 등 하드웨어 월렛에서 raw 데이터 직접 확인
3. **CI/CD 무결성 강화** — 서명된 커밋 강제, 배포 전 자동화 무결성 검사
4. **에어갭 서명 환경** — 멀티시그 서명 시 인터넷 격리 환경에서만 작업

### 교훈

Bybit 해킹은 **"보이는 것이 전부가 아니다"**라는 교훈을 준다. 공급망의 어느 한 고리만 무너져도 최종 서비스가 무너질 수 있으며, 국가 지원 해킹 조직의 위협은 단순한 사이버 범죄를 넘어선 **지정학적 안보 위협**이다.

## 3. 2024 BlackSuit 랜섬웨어 CDK Global 공격

### 사건 개요

2024년 6월 18일, 미국 자동차 딜러십 관리 소프트웨어(DMS) 업계 1위 기업인 **CDK Global**이 랜섬웨어 공격을 받아 전 시스템이 마비되었다. 공격 그룹은 **BlackSuit 랜섬웨어**로 확인되었으며, 미국·캐나다의 자동차 딜러십 **15,000개소**의 영업이 약 2주간 중단되었다. 총 피해액은 약 **10억 달러(한화 약 1조 4,000억 원)**으로 추산된다.

| 항목 | 내용 |
|------|------|
| 발생 일시 | 2024년 6월 18일 |
| 공격 그룹 | BlackSuit 랜섬웨어 (구 Royal → Conti 계열) |
| 피해 기업 | CDK Global (美 자동차 딜러십 소프트웨어 1위) |
| 피해 범위 | 미국·캐나다 자동차 딜러십 15,000개소 |
| 피해액 | 약 10억 달러 (랜섬 약 $25M 지불 포함) |
| 서비스 중단 기간 | 약 2주 |

### CDK Global이란?

CDK Global은 자동차 딜러십의 모든 업무를 통합 관리하는 **딜러십 관리 시스템(DMS)**을 제공하는 SaaS 기업이다. 차량 재고 관리, 판매 계약, 금융 조회, AS 예약, 고객 정보 관리를 모두 담당한다. 공격 기간 중 많은 딜러십이 차량 판매 계약서를 **손으로 직접 작성**해야 했다.

### 공격 경위

#### 1단계 — 사회공학으로 자격증명 탈취

BlackSuit은 IT 헬프데스크를 사칭하거나 **보이스 피싱(Vishing)**으로 CDK Global 직원을 속여 내부 시스템 자격증명을 탈취한 것으로 추정된다.

#### 2단계 — 권한 상승 및 횡적 이동

- **PsExec:** 원격 프로세스 실행을 통한 횡적 이동
- **Kerberoasting:** Active Directory 서비스 계정 패스워드 크래킹
- **Living-off-the-Land(LotL):** 정상 시스템 도구를 악용하여 탐지 회피

#### 3단계 — 이중 갈취 준비 (데이터 유출)

랜섬웨어 실행 전 내부 데이터를 외부로 대량 유출했다. **"암호화 + 유출 공개 협박"**을 조합한 이중 갈취(Double Extortion) 전술이다.

#### 4단계 — VM 내부에서 랜섬웨어 실행

호스트 EDR의 탐지를 피하기 위해 **가상머신(VM) 내부에서 랜섬웨어를 실행**했다. CDK Global 핵심 인프라가 암호화되며 전국 딜러십이 일제히 접속 불가 상태가 되었다.

#### 5단계 — CDK 사칭 2차 공격

사고 직후 CDK Global을 사칭하는 전화로 딜러십 직원들을 추가 기망하는 2차 공격까지 감행했다.

### BlackSuit 랜섬웨어 그룹의 정체

**Conti → Royal → BlackSuit**으로 이어지는 러시아·동유럽 기반 조직의 후계자다.

```
Conti (2020~2022, 러-우 전쟁 지지 후 해산)
    └→ Royal Ransomware (2022~2023)
           └→ BlackSuit (2023~현재)
```

- **RaaS 아님:** 외부 제휴 없이 직접 운영하는 폐쇄적 조직
- **이중 갈취:** 데이터 암호화 + 유출 협박 병행
- **주요 타깃:** 의료, 제조, 정부, 교육, 소매·유통

| 연도 | 피해 대상 | 주요 내용 |
|------|-----------|-----------|
| 2023 | Dallas(美 시) | 도시 IT 인프라 마비, 비상사태 선포 |
| 2024 | CDK Global | 딜러십 15,000개 마비, 약 $1B 피해 |
| 2025 | 국내 제조업체 등 | 다크웹 유출 협박 지속 |

### 근본 원인 분석

| 원인 | 내용 |
|------|------|
| 중앙집중식 SaaS의 단일 장애점 | 하나의 공격으로 수만 개 기업 동시 마비 가능 |
| 인적 보안 취약 | 사회공학 한 번으로 기술적 보안 전체를 우회 |
| 이상 행위 탐지 미흡 | Kerberoasting·대규모 횡적 이동 등 IOC를 조기 탐지 못함 |
| VM 기반 공격 대응 부재 | 하이퍼바이저 레벨 보안 모니터링 없음 |
| 백업 복구 전략 부재 | 2주 이상 복구 지연 = 오프라인 백업 체계 미비 |

### CDK Global의 대응

| 조치 | 내용 |
|------|------|
| 전체 서비스 오프라인 | 피해 확산 방지를 위해 즉시 차단 |
| 랜섬 지불 | 약 $25M 상당 비트코인 지불 추정 |
| 순차 복구 | 약 2주에 걸쳐 딜러십 시스템 단계적 복원 |
| 보안 강화 | 다중 인증, 접근 통제 강화 발표 |

### 법 집행 기관의 대응 — 오퍼레이션 체크메이트

2025년 7월 24일, 국제 법 집행 기관이 **오퍼레이션 체크메이트(Operation Checkmate)**를 통해 BlackSuit 인프라를 일부 해체했다. 서버 4대 압수, 도메인 9개 압류, 불법 암호화폐 100만 달러 이상 몰수.

### 미흡했던 점 및 보완 방향

1. **실전형 보안 인식 교육** — 헬프데스크 사칭 시나리오 등 정기 훈련 필요
2. **마이크로 세그멘테이션** — 내부 시스템 간 횡적 이동 차단
3. **하이퍼바이저 레벨 EDR/XDR** — VM 내부까지 모니터링 가능한 보안 솔루션 도입
4. **3-2-1 백업 원칙** — 3개 복사본, 2개 미디어, 1개 오프라인 백업 유지
5. **SaaS 공급업체 보안 감사** — 핵심 공급업체에 대한 보안 계약 조건 포함

### 교훈

CDK Global 사건은 **공급망 랜섬웨어 한 번으로 산업 전체가 마비될 수 있다**는 것을 보여준다. 랜섬 지불은 근본 해결책이 아니며, 공격을 완전히 막을 수 없다면 **빠른 복구 능력(Resilience)**이 예방 못지않게 중요하다.

## 4. 2026 유럽연합 집행위원회 AWS 클라우드 침해

### 사건 개요

2026년 3월 24일, 유럽연합 집행위원회(European Commission)의 공식 웹사이트 플랫폼 **europa.eu**를 호스팅하는 **AWS 클라우드 인프라**가 침해되었다. 공격자는 오픈소스 보안 스캐너 **Trivy**의 공급망 공격을 통해 AWS 자격증명을 탈취하여 **91.7GB**의 데이터를 유출했다. EU 내 **71개 기관**의 데이터가 영향을 받았으며, 이후 해커 그룹 **ShinyHunters**가 탈취 데이터를 다크웹에 공개했다.

| 항목 | 내용 |
|------|------|
| 발생 일시 | 2026년 3월 19일 (최초 침투) ~ 3월 28일 (다크웹 공개) |
| 공격 그룹 | TeamPCP (침투) + ShinyHunters (데이터 공개) |
| 피해 규모 | 91.7GB 압축 데이터, EU 71개 기관 |
| 공격 벡터 | Trivy 오픈소스 보안 스캐너 공급망 타협 |
| 유출 데이터 | 이름·이메일·이메일 내용, 이메일 파일 51,992건(2.22GB) |

### 공격 경위

#### 1단계 — Trivy 공급망 공격

**Trivy**는 컨테이너·클라우드 환경의 취약점을 스캔하는 널리 사용되는 오픈소스 보안 도구다. TeamPCP는 Trivy 패키지를 타깃으로 한 **공급망 공격(Supply Chain Attack)**을 통해, 이 도구를 사용하는 시스템에서 AWS API 키를 자동으로 수집하는 악성 버전을 배포했다.

유럽연합 집행위원회는 이 **침해된 버전의 Trivy**를 사용하고 있었고, 공격 당시 AWS 계정 관리 권한이 있는 API 키가 탈취되었다.

#### 2단계 — AWS 환경 침투 및 횡적 이동

2026년 3월 10일, TeamPCP는 탈취한 AWS API 키를 이용하여 유럽연합 집행위원회의 클라우드 환경에 침투했다.

- **TruffleHog** 도구를 활용하여 추가 시크릿(API 키, 자격증명)을 검색
- **신규 액세스 키를 기존 사용자에 연결**하여 탐지를 회피하는 방법으로 지속적 접근 확보
- 추가 정찰 활동 후 데이터 대량 유출

#### 3단계 — 데이터 유출 및 공개

2026년 3월 19~24일 사이 91.7GB(압축) 규모의 데이터를 외부로 유출했다. 3월 28일, ShinyHunters 그룹이 이 데이터를 **다크웹 유출 사이트에 공개**했다.

유출 데이터에는 다음이 포함되었다.
- EU 집행위원회 및 29개 이상 EU 기관 관계자의 이름·이메일
- 이메일 콘텐츠 파일 **51,992건** (일부 사용자 제출 내용 포함)
- 42개 EU 집행위원회 내부 클라이언트 관련 데이터

### GDPR의 아이러니

유럽연합은 **GDPR(일반개인정보보호법)**을 제정하여 전 세계 가장 엄격한 개인정보 보호 기준을 만든 주체다. 그 집행위원회가 자체 GDPR 기준을 충족하지 못하는 방식으로 데이터를 관리하다 침해당했다는 점에서 깊은 아이러니가 있다.

### 근본 원인 분석

| 원인 | 내용 |
|------|------|
| 오픈소스 의존성 검증 부재 | 침해된 Trivy 버전을 무결성 검증 없이 사용 |
| AWS 자격증명 과도한 권한 부여 | 관리 권한이 있는 API 키가 탈취될 경우 전체 계정 장악 가능 |
| 시크릿 탐지 체계 미비 | TruffleHog으로 추가 자격증명 수집되는 동안 탐지 못함 |
| 클라우드 로그 모니터링 미흡 | 비정상 API 키 생성·사용 패턴을 실시간 탐지하지 못함 |

### 유럽연합의 대응

- 2026년 3월 27일 **공식 침해 사실 공표** (3일 만에 공개, 상대적으로 신속)
- CERT-EU가 사고 원인·범위·영향을 분석하여 **4월 3일 상세 보고서 발표**
- 영향받은 71개 기관에 통보 및 추가 피해 방지 조치
- Trivy 및 기타 오픈소스 도구 사용 환경 점검 권고

### 미흡했던 점 및 보완 방향

1. **오픈소스 패키지 무결성 검증** — 해시 서명 확인, SBOM(소프트웨어 자재명세서) 관리 의무화
2. **최소 권한 원칙 (PoLP)** — API 키에 꼭 필요한 권한만 부여, 관리 권한 키 사용 최소화
3. **시크릿 관리 솔루션** — AWS Secrets Manager, HashiCorp Vault 등으로 자격증명 중앙 관리
4. **CloudTrail·GuardDuty 활성화** — AWS 내 비정상 API 호출·신규 키 생성을 실시간 알림
5. **SBOM 기반 공급망 보안** — 사용 중인 모든 오픈소스 컴포넌트를 추적하고 침해 여부 모니터링

### 교훈

유럽연합 집행위원회 침해 사건은 **"신뢰할 수 있다고 여겨지는 보안 도구 자체가 공격 벡터가 될 수 있다"**는 것을 보여준다. 오픈소스 소프트웨어 공급망 공격은 2026년 가장 빠르게 증가하는 공격 트렌드 중 하나다.

## 5. 2026 Telus 캐나다 통신사 700TB 데이터 탈취

### 사건 개요

2026년 3월, 캐나다 주요 통신사 **Telus**가 사이버 침해 사고를 확인했다. 해킹 그룹 **ShinyHunters**는 자신들이 **700TB** 이상의 데이터를 탈취했다고 주장했으며, 여기에는 고객 개인정보, 통화 데이터, 신원조회 정보, 소스코드 등이 포함된다.

| 항목 | 내용 |
|------|------|
| 발생 일시 | 2026년 3월 |
| 공격 그룹 | ShinyHunters |
| 탈취 주장 규모 | 700TB 이상 |
| 유출 데이터 | 고객 PII, 통화 기록, 신원조회 데이터, 소스코드 |
| 초기 침투 경로 | Salesloft 플랫폼 침해로 탈취한 Telus Google Cloud 자격증명 |

### 공격 경위

#### 초기 침투 — 연쇄 공급망 공격

이 사건의 핵심은 **공급망 연쇄 침해(Cascading Supply Chain Attack)**다.

1. ShinyHunters는 2025년에 **영업 자동화 플랫폼 Salesloft**를 먼저 침해했다.
2. Salesloft 데이터베이스에서 Telus가 사용하던 **Google Cloud Platform(GCP) 자격증명**을 발견했다.
3. 탈취한 GCP 자격증명으로 **Telus 내부 클라우드 환경에 접근**했다.

#### 데이터 탈취

클라우드 접근 권한을 확보한 후, 공격자는 다음 데이터를 대량 유출했다.

- 고객 성명, 주소, 전화번호 등 **PII(개인식별정보)**
- **통화 데이터 레코드(CDR):** 누가 누구에게 언제 얼마나 통화했는지 기록
- **신원조회(Background Check) 데이터:** Telus가 직원 채용 시 수집하는 민감 정보
- **소스코드:** Telus 내부 시스템의 소프트웨어 코드

#### 협박 및 공개

Telus가 랜섬 지불을 거부하자, ShinyHunters는 탈취 데이터를 다크웹에 공개하거나 판매하겠다고 협박했다.

### ShinyHunters 그룹이란?

ShinyHunters는 2020년부터 활동 중인 데이터 탈취·갈취 전문 해커 그룹으로, 전통적인 랜섬웨어(암호화) 없이 **순수 데이터 유출 협박** 방식을 주로 사용한다.

| 연도 | 피해 대상 | 내용 |
|------|-----------|------|
| 2024 | Ticketmaster | 5억 6천만 명 데이터 탈취 |
| 2026년 1월 | Crunchbase | 200만 건 이상 유출 |
| 2026년 3월 | Telus | 700TB 탈취 주장 |
| 2026년 3월 | 유럽연합 집행위원회 | EU 침해 데이터 다크웹 공개 |

### 근본 원인 분석

| 원인 | 내용 |
|------|------|
| 3rd 파티 SaaS 자격증명 관리 실패 | Salesloft에 GCP 자격증명을 저장하다 연쇄 침해 |
| 클라우드 자격증명 순환 미흡 | 장기간 유효한 정적 자격증명을 사용 |
| 데이터 분류·최소화 미흡 | 단일 클라우드 환경에 과도하게 많은 민감 데이터 집중 |
| CDR 데이터 보호 부족 | 개인 통신 패턴을 담은 고민감도 데이터의 접근 통제 미비 |

### Telus의 대응

- 사고 사실 공식 확인 및 조사 착수
- 영향받은 고객에게 통보 및 모니터링 서비스 제공
- 클라우드 환경 자격증명 전면 재발급
- 3rd 파티 플랫폼에 저장된 자격증명 전수 점검

### 미흡했던 점 및 보완 방향

1. **3rd 파티 SaaS에 자격증명 저장 금지** — GCP·AWS 키를 외부 플랫폼에 평문 보관하는 관행 즉시 중단
2. **단기 수명 자격증명(Short-lived Credentials)** — 정적 API 키 대신 IAM Role 기반 임시 토큰 사용
3. **데이터 최소화(Data Minimization)** — 불필요한 데이터를 보관하지 않고, 민감 데이터는 계층적으로 분리 저장
4. **CDR 암호화 및 접근 감사** — 통화 기록 등 고민감도 데이터는 암호화 + 접근 로그 실시간 감사
5. **공급업체 보안 평가** — Salesloft 등 연동 SaaS 플랫폼의 보안 수준 정기 평가

### 교훈

Telus 사건은 **"내 시스템이 직접 뚫리지 않아도, 내가 믿는 파트너가 뚫리면 나도 뚫린다"**는 연쇄 공급망 리스크를 보여준다. 수백 개의 SaaS와 연동되는 현대 기업 환경에서 3rd 파티 자격증명 관리는 더 이상 선택 사항이 아니다.

## 6. 2026 UMMC 의료센터 Medusa 랜섬웨어 공격

### 사건 개요

2026년 2월 19일, 미국 미시시피주 최대 의료 기관인 **미시시피 의과대학 의료센터(UMMC, University of Mississippi Medical Center)**가 랜섬웨어 공격을 받아 주 전역 **35개 클리닉이 전면 폐쇄**되었다. **Epic EHR(전자의무기록) 시스템이 9일간 오프라인** 상태로 유지되어 예약·수술·영상검사가 전면 취소되었으며, 의료진은 환자 기록을 수기로 작성해야 했다. 3월 12일, **Medusa 랜섬웨어 그룹**이 공식 책임을 인정하며 80만 달러를 요구했다.

| 항목 | 내용 |
|------|------|
| 발생 일시 | 2026년 2월 19일 |
| 공격 그룹 | Medusa 랜섬웨어 |
| 피해 기관 | UMMC (미시시피주 최대 의료센터) |
| 피해 범위 | 주 전역 클리닉 35개소 전면 폐쇄 |
| Epic EHR 중단 | 9일간 오프라인 |
| 요구 랜섬 | $800,000 (3월 20일까지 미지불 시 데이터 공개 협박) |

### 의료 현장의 실제 피해

랜섬웨어는 단순한 데이터 유출이 아닌 **환자 안전에 직접적인 위협**을 초래했다.

- 예약 환자 **수천 명의 진료 취소**
- 선택적 수술·비응급 시술 전면 중단
- **영상 검사(MRI·CT 등) 중단** — 진단 지연
- 의료진이 컴퓨터 없이 **수기 차트**로 환자 정보 기록
- 일부 중증 환자를 **타 의료 기관으로 전원(이송)**

### 공격 경위

#### 1단계 — 초기 침투

Medusa 그룹은 주로 다음 방법으로 의료 기관에 침투한다.
- **피싱 이메일**을 통한 자격증명 탈취
- **VPN·원격 접속 솔루션의 취약점** 악용
- **RDP(원격 데스크톱 프로토콜) 무차별 대입** 공격

#### 2단계 — 내부 확산 및 정찰

침투 후 Active Directory를 통해 병원 전체 네트워크로 확산하며 중요 시스템 위치를 파악했다. 특히 **Epic EHR 서버** 등 운영 핵심 시스템을 우선 타깃으로 삼았다.

#### 3단계 — 이중 갈취

랜섬웨어 실행 전, 환자 정보·연구 데이터·내부 문서를 **외부로 유출**했다. 이는 "암호화 복구 불가 시에도 데이터 공개"라는 이중 압박을 가하기 위한 전술이다.

#### 4단계 — 랜섬웨어 배포

Epic EHR을 포함한 핵심 IT 인프라 전체를 암호화하여 의료 운영을 완전히 마비시켰다.

### Medusa 랜섬웨어 그룹이란?

Medusa는 2023년부터 본격 활동을 시작한 **RaaS(랜섬웨어-as-a-Service)** 그룹으로, 의료·교육·정부 기관을 집중 공략하는 것으로 알려져 있다.

- **이중 갈취 전문:** 데이터 암호화 + 유출 협박 병행
- **협상 공개:** Medusa 블로그에서 피해 기관과의 협상 내용을 공개하여 추가 압박
- **짧은 데드라인:** 통상 7~14일 내 지불 요구, 불응 시 즉시 데이터 공개

### 왜 의료 기관이 최우선 표적인가

| 이유 | 내용 |
|------|------|
| 데이터 가치 | 의료 기록은 다크웹에서 금융 데이터보다 10배 높은 가격에 거래 |
| 운영 긴급성 | 병원은 시스템 중단이 곧 환자 생명 위협 → 빠른 지불 압박 |
| 보안 투자 부족 | 의료 기관은 IT 보안 예산 부족, 레거시 시스템 비율 높음 |
| 규제 이중 압박 | HIPAA 등 규제 위반 시 추가 과징금 → 이중 압박 |

### 근본 원인 분석

| 원인 | 내용 |
|------|------|
| 레거시 시스템 및 취약한 원격 접속 | VPN·RDP 취약점이 패치되지 않은 상태 |
| 의료 환경의 보안 투자 부족 | 환자 치료 우선으로 보안 예산이 상대적으로 적음 |
| EHR 시스템의 단일 장애점 | Epic이 다운되면 전 클리닉이 동시에 마비 |
| 오프라인 백업 미비 | 9일간 복구 지연 = 신속 복구 가능한 백업 체계 부재 |
| 이상 행위 탐지 부재 | 내부 확산 및 데이터 유출 단계에서 탐지 실패 |

### UMMC의 대응

- 공격 인지 즉시 **전체 시스템 오프라인** 전환, 피해 확산 차단
- 주정부 및 연방 당국(FBI·CISA)에 신고 및 수사 협조
- 영향받은 35개 클리닉 **순차적 복구** 진행
- 환자 진료 연속성 유지를 위한 **응급·우선순위 기반 수기 운영 체계** 가동

### 미흡했던 점 및 보완 방향

1. **의료 전용 사이버보안 예산 확보** — HIPAA 규정 준수를 위한 최소한의 보안 투자가 필요
2. **EHR 오프라인 백업 및 DR 계획** — Epic 등 핵심 시스템의 오프라인 백업과 복구 절차 정기 훈련
3. **네트워크 세그멘테이션** — EHR 시스템을 격리된 네트워크 세그먼트에 배치하여 횡적 확산 차단
4. **원격 접속 강화** — VPN·RDP에 MFA 필수 적용, 접근 허용 IP 화이트리스트 관리
5. **의료진 보안 인식 교육** — 의사·간호사 등 의료진 대상 피싱 대응 교육

### 교훈

UMMC 랜섬웨어 사건은 **사이버 공격이 생명을 위협하는 물리적 재해로 이어질 수 있다**는 것을 실증했다. 의료 분야의 디지털화가 가속될수록, 사이버 보안은 선택이 아닌 환자 안전의 필수 요소다. 랜섬웨어 2026 통계에 따르면 의료 부문 공격은 전년 대비 30% 이상 증가했으며, 이 추세는 계속될 전망이다.
